# 03 · Proposed canonical schema

**Org-centric model.** Following the Brendan/Diego agreement: `companies` is the **universal org table** ("companies needs to be orgs"). Every football entity we track or sell to — club, league/competition, federation, national team, broadcaster/rights-holder, agency, media, service, school — is an **org** with one or more **types**, and org-to-org relationships model the "weird web" (nesting, governance, broadcast rights). GPS integration is **standardization** (shared vocabulary + the `gps_org_id` link), **not** a structural merge; the CRM keeps the shape useful to selling.

Groups: **(I)** reused Atomic · **(II)** extended Atomic · **(III)** new domain. PK = `bigint generated by default as identity` unless noted. `created_at`/`updated_at` where applicable. Status fields stay free text until inventoried (doc 05 §J). See doc 07 for GPS/standardization.

---

## I. Reused Atomic tables (no structural change)

`sales`, `tags`, `contact_notes`, `deal_notes`, `configuration`, `favicons_excluded_domains`. `configuration`: change data only (GBP, title, logos; `dealStages` no longer the stage source of truth). `contact_notes`/`deal_notes` = free-text store, coexist with `activities`. **Auth = Google OAuth on Supabase** (decision A); `sales` stays as-is, **no `entra_oid`** — CRM and GPS are separate databases.

## II. Extended Atomic tables

### companies — the universal org

Any football org (and generic companies). Both prospects and the broader football world model live here, so all of them can be deal targets.

- **New columns:**
  - `kind text` — the org's **primary/display type** (quick filter). Source of truth for the full set is `company_org_types` (N–N); `kind` is a denormalized mirror of the primary type.
  - `ltv bigint`, `updated_at timestamptz not null default now()`.
  - `coda_row_id text unique` (generic, usually null), `legacy_ref text unique` (vibecoded `a##`/`svc##`).
  - `slug text unique` (nullable) — standardized handle, aligned with GPS conventions.
- **Reuses:** name, website (citext), country, city, linkedin_url, logo (jsonb), sales_id, created_at, sector/size/phone/address/etc.
- **PK** id · **FK** sales_id→sales · **Indexes** companies_kind_idx, companies_name_idx.

### contacts — people (staff, decision makers, agents, coaches)

- `company_id` kept for Atomic compat; **source of truth for employer = `company_contacts`** (decision 4); `contacts.company_id` mirrors the primary+current link. A person can hold several org links (e.g. a coach at a club **and** a national team).
- **New:** `updated_at`, `coda_row_id text unique`, `legacy_ref text`.
- **Reuses:** first/last_name, title, email_jsonb, phone_jsonb, linkedin_url, tags, status, background, gender, sales_id, etc.

### deals — multi-brand opportunities (against any org)

`deals.company_id → companies` — since leagues, federations, national teams and broadcasters are now orgs, **deals can target any of them**, not just clubs (Brendan: "treating all of the leagues and other things as their own entities that we are going to target for sales").

- **New:** `brand_id` → brands (NOT NULL after backfill); `stage_id` → pipeline_stages (source of truth); `probability smallint` CHECK 0–100; `coda_row_id`, `legacy_ref`.
- **brand↔stage integrity (decision 2):** `pipeline_stages UNIQUE (brand_id, id)` + composite FK `deals(brand_id, stage_id) → pipeline_stages(brand_id, id)`.
- `deals.stage` slug = temporary Atomic-Kanban compat, synced from `stage_id`.
- **Reuses:** name, company_id, contact_ids (bigint[]), category, description, amount, expected_closing_date, sales_id, index, archived_at, created/updated.
- No unique on (company_id, brand_id) (decision 11). Weighted value derived, not persisted.

### tasks — tasks and follow-ups

- `contact_id` → **nullable**; optional FKs `deal_id`, `company_id`, `activity_id`. (No separate `league_id` — a league is a `company`.)
- CHECK `num_nonnulls(contact_id, deal_id, company_id, activity_id) >= 1`.
- `activity_id` nullable + partial unique → an activity spawns at most one follow-up.

## III. New domain tables

### org_types  ·  company_org_types  *(multi-type orgs)*
- `org_types` — lookup: id, `code text unique`, name. Seed (standardized with GPS vocabulary, extensible): club, league, competition, federation, national_team, broadcaster, rights_holder, agency, media_outlet, service, school, university, grassroots, company. Brendan: "continue to add to that org-type table."
- `company_org_types` — junction: PK (company_id, org_type_id); cascade. **Source of truth for an org's types** (an org can be several — e.g. a body that is both a league and a federation). `companies.kind` mirrors the primary one.

### org_relationships  *(the "weird web": nesting, governance, broadcast rights)*
- id, `from_company_id` → companies, `to_company_id` → companies, `relationship_type text` CHECK in (parent_of, governs, broadcasts, affiliated_with, owns), `region text` (nullable — for rights-holders split geographically), `is_current bool default true`, start_date, end_date, created_at.
- Models: **nested leagues** (`parent_of`, per-country pro+youth pyramid), **federation → league / national team** (`governs`), **rights-holder/broadcaster → competition** (`broadcasts`, with `region` for the geo-split), club affiliations, etc.
- Unique partial `(from_company_id, to_company_id, relationship_type, region) where is_current`. Self-referential; both ends are orgs.

### competitions — 1:1 extension for leagues/competitions
- `company_id bigint primary key references companies(id) on delete cascade` (the league-org).
- `level text` (tier), `format text`, `season text` (nullable). Country comes from `companies.country`; the pyramid/parent comes from `org_relationships`.
- *(A league is an org of type `league`/`competition`; this holds its competition-specific attributes.)*

### nations *(lookup)*
id, `code text unique`, name. Used by `clubs.nation_id` and `player_citizenships`. Real codes are dirty (Aberdeen=ENG) → clean on import.

### clubs — 1:1 extension for clubs
- `company_id bigint primary key references companies(id) on delete cascade` (PK = FK, 1:1).
- `league_company_id bigint references companies(id)` — the competition the club plays in (a league-**org**, not a lookup).
- `nation_id → nations`, `crest text`, `keeper_count smallint default 0`, `contract_end date`, `hit_list boolean default false`.
- `xg_status text` — no CHECK (all `Pending` in export).
- `email_domain text`, `email_pattern text`, `notes text`.
- `coda_row_id text unique` (the club's Coda id).
- `gps_org_id bigint unique` — **nullable & optional** reconciliation link to the GPS tenant. Kept as a low-cost soft link: filled by hand on deal-won *if* the team finds it useful, left null otherwise. CRM and GPS are separate systems with different purposes/hosts, so we don't force a one-to-one tie (doc 05 §0, doc 07).
- Indexes clubs_league_idx, clubs_xg_status_idx, clubs_gps_org_idx.

### brands  ·  pipeline_stages  ·  company_brands
- `brands` (id, key unique, name, accent_color, is_active). Seed: xg #FF0092, onesport #9B1B30, calma #CEAD54, gkcom #00F07B.
- `pipeline_stages` (id, brand_id → brands, position, name, slug, default_probability CHECK 0–100, is_won, is_lost). Unique (brand_id, position), (brand_id, slug), **UNIQUE (brand_id, id)** (enables the composite FK). Seed: xG 8 / Onesport 6 / Calma 6 / gkcom 6.
- `company_brands` (N-N): PK (company_id, brand_id); cascade.

### company_contacts — N-N person↔org (role, relationship, history)
- id pk, company_id → companies (cascade), contact_id → contacts (cascade).
- `role text CHECK in (sporting_director, head_of_recruitment, gk_coach, head_coach, analyst, scout, manager, ceo, sports_secretary, agent, other)` — includes coach/agent roles; an org being a national team or agency is fine since it's just another org.
- `title text`; `relationship_status text` (no CHECK; Very/Moderate/Limited Contactable); `preferred_contact bool`; `instagram_status`/`linkedin_status text` (no CHECK; normalize case); `notes text`; `is_primary bool default false` (defines contacts.company_id); `verified_at date`; `start_date`/`end_date`; `is_current bool default true`; created/updated.
- **Integrity (decisions 4,5):** no global unique; partial unique `(company_id, contact_id, role) where is_current`; single active primary `unique (contact_id) where is_primary`. CHECKs: active link has no end_date; coherent dates; primary ⇒ current. History allowed. *(A coach at a club and a national team = two rows here; that's the multi-org case.)*

### players
- id pk, full_name, first_name, last_name. `date_of_birth date` (preferred) / `age_imported smallint` (no DOB in source). `status text` (derivable from the 8 funnel steps, doc 05 §D). `coda_row_id text unique`, created/updated.
- League **not stored** (derive from the current club). Funnel = **8 boolean columns** (`step_follow_back` … `step_contract`) + derived `status` (decision D).

### player_org_assignments — N-N player↔org (clubs **and** national teams)
- id pk, player_id → players (cascade), `org_id bigint references companies(id)` — the org (club or national team).
- `relationship_type text not null CHECK in (owned, current, loaned_in, loaned_out, target, national_team)` — non-exclusive rows (Brendan's case: owned by club A, loaned at club B, plus a national-team call-up). App/CHECK keeps `org_id` to a club or national_team org.
- `is_current bool default true`, start_date, end_date, created_at. No global unique; partial unique `where is_current`; date CHECKs.
- A **`current_squad` view** materializes "physically at the club now" = `(owned ∧ ¬loaned_out) ∨ loaned_in` from these rows (decision C) — raw rows stay faithful to Coda, the view gives the coherent read.

### player_citizenships — N-N player↔nation
- PK (player_id, nation_id); cascade. Covers Brendan's "a player can have multiple citizenship."

### player_representations — agent (person) + agency (org)
- id, player_id → players (cascade).
- `agency_company_id bigint references companies(id)` — the agency as an **org** (type `agency`), nullable.
- `agent_contact_id bigint references contacts(id)` — the agent as a **person**, nullable (linked to the agency via `company_contacts.role='agent'`).
- `rep_code text` (RL/ZY…), `agent_expiry date` (normalize dirty Coda formats), `is_current bool default true`, start/end, created_at.
- *(Replaces the old free-text `agency_name`; agents and agencies are now first-class — Brendan: "agent is a person, agency is an org.")*

### contact_social_profiles  ·  player_social_profiles
- `contact_social_profiles` (id, contact_id → contacts, `platform CHECK in (instagram, x, tiktok, other)`, url, created_at; unique (contact_id, platform, url)). LinkedIn stays in `contacts.linkedin_url`.
- `player_social_profiles` — same shape with player_id (confirm format; empty in sample).

### deal_modules (xG)
PK (deal_id, module); `module CHECK in (ongoing_support, transfer_market_support, opposition_analysis, api)`; cascade. xG-brand deals only.

### activities (unified outreach)
- id pk, type, channel, status, note, occurred_at, sales_id → sales.
- Nullable targets: `company_id`, `contact_id`, `deal_id`, `player_id`. (No `league_id` — league outreach targets the league-org via `company_id`.)
- CHECK `num_nonnulls(company_id, contact_id, deal_id, player_id) >= 1`. Spawns at most one `tasks` follow-up. Extend `activity_log` with a UNION of activities.

### target_lists  ·  org_target_lists  ·  player_target_lists
- `target_lists` (id, name, kind org/player/mixed).
- `org_target_lists` PK (target_list_id, company_id) — any org (club, league, etc.) can be on a list.
- `player_target_lists` PK (target_list_id, player_id).
- Today: `Hit List` → `clubs.hit_list`; `Target Players` → `player_org_assignments` type `target` (empty in export). Named lists by requirement.

### teams / sports / team_sports (Onesport)
teams (id, company_id → companies, name, `level CHECK in (grassroots, school, university, club)`, created/updated). sports (id, name unique). team_sports PK (team_id, sport_id). company 1–N teams; team N–N sports.

**Cross-cutting (react-admin):** each new table = a `<Resource>` + full-text; keep PK `bigint identity`; update `*_summary` views; keep native arrays (tags, contact_ids). **Per-table RLS (decision 10):** RLS + policies + grants in each create migration; single-tenant at launch, RBAC at M6 (GPS 4-layer pattern as template — doc 07).

**Standardization with GPS (doc 07):** the org-type vocabulary, the `slug`/timestamp/soft-delete conventions, and `bigint` identity PKs are aligned with GPS so both systems "speak the same way." Reconciliation stays the `clubs.gps_org_id` link — no shared tables.
